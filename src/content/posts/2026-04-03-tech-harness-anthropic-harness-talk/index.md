本篇记录一下anthropic对于harness工程的讲解。

```
来源:https://www.anthropic.com/engineering/harness-design-long-running-apps
```

Anthropic提到，两个尝试一个是如何设计更好的前端样式，以及如何在没有人类介入情况下构建完整应用（最终产生了[frontend design skill](https://github.com/anthropics/claude-code/blob/main/plugins/frontend-design/skills/frontend-design/SKILL.md)和[long-running coding agent harness](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)）。

作者从GAN中获得灵感，我们都知道GAN的核心两个结构就是 **generator**和 **evaluator**,因此他的思想就是构建这样两个agents。*评估器还有一个要求就是必须要把主观的判断转换为能够具体稳定评分的条目*。

作者把这个技术应用到了`long-running coding agent harness` 中，另外还加上了之前的harness经验:

+ 把构建过程拆分成可处理的小块。
+ 用结构化工件在不同会话之间交接上下文。

> **最终得到的结果是一个由规划器、生成器和评估器组成的三智能体架构，它可以在持续数小时的自主编程会话中产出内容丰富的全栈应用。**

## 两个场景的实验

作者主要在前端和全栈编程都做了实验。

### 前端实验

前端实验中提到，虽然对于审美每个人都是主观的，但是可以把 `设计原则和审美偏好` 输入进去，然后分解处几个评分标准。

作者在定义好这些评分后，在具体的搭建上，基于 [Claude Agent SDK](https://platform.claude.com/) 搭建了整个循环，使编排保持相对直接。生成器智能体会先根据用户提示创建一个 HTML/CSS/JS 前端。我给评估器接入了 **Playwright MCP**，这样它就能在打分前直接操作实时页面。实际运行中，评估器会自己浏览页面、截图，并仔细查看实现，再输出评分和详细批评意见。这些反馈会作为下一轮迭代的输入返回给生成器。

生成器在每轮评估后做一个策略决策：如果分数趋势良好，就沿当前方向继续打磨；如果方向不行，就彻底切换到另一种审美路线。文中提到，跨多次运行来看，评估器给出的评分通常会随着迭代上升，之后进入平台期，而仍有进一步提升空间。有些生成是渐进式优化，另一些则会在迭代之间出现非常明显的审美转向。

> 评分标准的表述方式，会以我原本没有完全预料到的方式影响生成器。比如加入“最好的设计应当达到博物馆级别”这样的措辞，会让设计朝某种特定视觉风格收敛。这说明，与这些标准绑定的提示语言会直接塑造输出的气质。
>
> 虽然分数整体上通常随迭代提高，但其模式并不总是线性递增。后期实现整体上往往更好，但我们可能也经常会更喜欢中间某一轮，而不是最后一轮。随着轮次增加，实现复杂度通常也会上升，因为生成器会在评估器反馈的推动下尝试更有野心的方案。

### 全栈实验

作者提出，*这种受 GAN 启发的模式应用到全栈开发中。生成器-评估器循环与软件开发生命周期天然契合，在这个过程中，代码审查和 QA 所扮演的结构性角色，与设计评估器是同一类东西。*

在这项工作中，作者在原始harness基础上构建了三智能体系统:
```
规划器（Planner）：我们之前的长时间 harness 要求用户一开始就提供详细规格。我想把这一步自动化，于是创建了一个规划器智能体，它接受一个仅有 1 到 4 句的简单提示，并将其扩展成完整的产品规格。我在提示词中要求它在范围上更有野心，同时专注于产品上下文和高层技术设计，而不是细粒度的技术实现。

这样做是因为我担心：如果规划器一开始就试图写出非常细的技术细节，但其中有错误，那么这些错误会在后续实现中不断放大。更明智的方式似乎是：约束智能体关注应该交付什么，而让它们在执行过程中自己找实现路径。我还要求规划器在产品规格中寻找嵌入 AI 功能的机会。（示例见文末附录。）

生成器（Generator）：早期 harness 的“一次只做一个功能”方法在范围管理上效果很好。我在这里也沿用了类似思路，让生成器按 sprint 工作，每次从规格中拿一个功能块来做。每个 sprint 使用 React、Vite、FastAPI 和 SQLite（后期改为 PostgreSQL）技术栈实现应用；同时要求生成器在每个 sprint 结束时先自评，再交给 QA。它还接入了 git 做版本控制。

评估器（Evaluator）：早期 harness 产出的应用往往看起来很惊艳，但你真正用起来时仍会发现不少 bug。为了解决这一点，评估器通过 Playwright MCP 像真实用户一样点击、操作正在运行的应用，测试 UI 功能、API 端点和数据库状态。随后，它会结合自己发现的 bug，以及一组借鉴前端实验、但适配为“产品深度、功能性、视觉设计、代码质量”的评分标准，对每个 sprint 进行评分。
```

这部分总体来说，表达了重点就是，现代的harness需要在开始前把任务进行规格化(Planner)，生成器每次只从规格中拿一个功能块做，并且要有能测试和自评的设计。这样的设计，保证生成器会提出自己要构建什么、如何验证成功；评估器则审查这个提案，确认生成器做的是正确的事情，这样两者就会反复迭代，直到达成一直。

## what next

作者文中中还提到了一点，就是随着模型的持续进步，它们能够工作得更久，并处理更复杂的任务。在某些情况下，这意味着围绕模型搭建的支架会随着时间推移变得不那么重要，开发者只要等下一代模型出现，就会发现某些问题自己消失了。

比如说之前提到的`上下文焦虑问题`，在之前还需要特殊的设计，但是随着模型的进度，逐渐变得不需要了。还有就是上面说的sprint机制，此前，sprint 结构之所以有帮助，是因为它能把工作拆成更小的块，让模型更连贯地推进。考虑到 Opus 4.6 的提升，很有理由相信模型已经可以原生处理这类任务，而不再需要这种程度的拆分。

再说一下评估器和生成器。作者提到，随着模型能力增强，评估器对于不同运行的“承重程度”也发生了变化，它是否有价值，取决于任务相对模型原生能力处于哪个位置。在 4.5 上，这个边界离得很近：我们的构建任务就处在生成器单独完成能力的边缘，因此评估器在整个构建过程里能捕捉到很多真实问题。

而在 4.6 上，模型的原始能力更强，这条边界就向外移动了。以前那些必须依赖评估器才能做稳的任务，现在很多已经落入生成器单独也能处理好的范围之内；对于这些任务，评估器就成了额外开销。但对于那些依然处在生成器能力边缘的部分，评估器仍然能带来真实提升。**评估器并不是一个固定的“要”或“不要”的选择。只有当任务超出当前模型单独可靠完成的范围时，它的成本才是值得的。**

所以, harness是不必要的吗？我们只需要等待模型不断的进步，就能把所有的harness去掉不是吗？我认为这样的想法过于极端，学习harness工程，能让我们面对模型能力不同的场景，都有能力去选择，如何基于`模型`这个地基，在上面添加合适的砖瓦。
